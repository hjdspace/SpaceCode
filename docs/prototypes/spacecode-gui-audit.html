设计系统一致性

SpaceCode 拥有四套精心设计的主题（Light / Dark / Anthropic / Anthropic Dark），每套包含 70+ CSS 变量。但新旧两套令牌系统共存导致了视觉不一致。

P0 双令牌系统共存导致视觉断裂
旧令牌系统（--bg-primary、--border-color、--bg-hover）与新令牌系统（--surface-border、--surface-glass-hover、--radius-sm）同时存在于 _variables.scss 中，不同组件引用不同的令牌集，导致同一应用内视觉风格不统一。

旧令牌（MCP/Skills 组件）
background: var(--bg-primary);
border: 1px solid var(--border-color);
border-radius: 8px; /* 硬编码 */
color: var(--text-primary);
新令牌（SCM 组件）
background: var(--surface-glass);
border: 1px solid var(--surface-border);
border-radius: var(--radius-lg);
color: var(--fg-primary);
P0 硬编码颜色值绕过了主题系统
多处组件使用了硬编码的十六进制颜色值，导致切换主题时这些元素无法跟随变化。SCM 的提交按钮使用 #6ecf9e 而非 var(--success)；MCP 组件的错误状态使用 #dc3545 而非 var(--error)；Agent badge 使用 rgba(99, 102, 241, ...)。
src/components/scm/ScmPanel.vue src/components/mcp/ConfigEditor.vue src/components/layout/ChatPanel.vue
P1 按钮样式在 4+ 组件中重复定义
.btn 类在 ConfigEditor.vue、McpManager.vue、McpServerEditor.vue、SkillsManager.vue 中各自独立定义，padding、font-size、border-radius 值各不相同。应提取为共享组件或全局 SCSS 类。
src/components/mcp/ConfigEditor.vue src/components/mcp/McpManager.vue src/components/mcp/McpServerEditor.vue src/components/skills/SkillsManager.vue
P1 过渡 (transition) 值硬编码不一致
设计令牌已定义 --transition-fast (150ms)、--transition-normal (250ms)，但部分组件仍使用 transition: all 0.2s ease 等硬编码值，导致动画节奏不统一。
P1 模态框模式不统一
McpManagerModal 使用简单 overlay + close 模式，McpServerEditor 使用 header/body/footer 对话框模式，Skills 的 CreateSkillDialog 又是另一种布局。缺乏统一的 AppModal / AppDialog 基础组件。

维度二

交互体验

SpaceCode 实现了丰富的交互模式（斜杠命令面板、@ 文件提及、内容可编辑区域的内联芯片），但部分核心交互存在可维护性和可用性风险。

P0 ChatInput.vue 超过 2000 行，承载过多职责
该单文件组件同时处理斜杠命令、文件提及、模型选择、Agent 选择、附件管理、拖放上传、提示词优化、ContentEditable 管理、粘贴处理等至少 9 项独立功能。这严重影响了可维护性和可测试性。应拆分为 3-5 个 composables 和子组件。
src/components/chat/ChatInput.vue (~2000 lines)
当前：单一巨型组件
// ChatInput.vue - 全部功能耦合在一起
const handleInput = () => {
  // 斜杠命令检测
  // @ 文件提及触发
  // ContentEditable 同步
  // 芯片渲染
  // ... 200+ 行逻辑
}
建议：职责拆分
// composables/
useSlashCommands.ts   // 斜杠命令逻辑
useFileMentions.ts    // @ 文件提及
useContentEditable.ts // 编辑器管理
useAttachments.ts     // 附件与拖放
usePromptOptimizer.ts // 提示词优化
P1 面板 resize 手柄缺少键盘替代方案
App.vue 和 ScmPanel.vue 中的面板宽度/高度调节仅支持鼠标拖拽（mousedown → mousemove → mouseup），没有键盘快捷键替代。无法使用鼠标操作的用户无法调整面板尺寸。
P1 自定义下拉菜单缺少 ARIA 角色标注
斜杠命令面板、上下文菜单、模型选择器、附件菜单等自定义弹出组件手动管理焦点，但未实现 role="listbox"、role="option"、aria-expanded、aria-activedescendant 等必要的 ARIA 属性。
P1 跨组件通信使用 CustomEvent 事件总线
大量使用 window.dispatchEvent(new CustomEvent(...)) 进行跨组件通信（如 'open-skills-manager'、'session-created'、'split-screen-requested' 等），这种模式脆弱且难以追踪。应改用 Pinia store 或 provide/inject 模式。
P2 面板宽度使用内联样式而非 CSS 变量
App.vue 通过 :style="{ width: ... }" 动态设置侧边栏和信息面板宽度，而非使用 CSS 自定义属性。这导致宽度变化无法通过 CSS transition 实现平滑动画。
P2 缺少桌面级响应式断点处理
作为桌面 Electron 应用，仅在 MessageItem.vue 中有一个 @media (max-width: 768px) 查询。当用户将面板 resize 到很窄的宽度时，缺乏对应的自适应布局。
维度三

可访问性

可访问性是 SpaceCode 当前最薄弱的环节。色彩对比度、键盘导航和屏幕阅读器支持均存在明显缺口。

色彩对比度分析

以下列出在 Light 主题下对比度不足的颜色组合（WCAG AA 标准要求正文文本 ≥ 4.5:1，大号文本 ≥ 3:1）：

颜色组合 前景色 背景色 对比度 用途 状态
--text-muted / --bg-primary Aa var(--swatch-light-bg) 3.8:1 小文本（14px） 不通过 AA
--text-disabled / --bg-primary Aa var(--swatch-light-bg) 2.5:1 禁用状态文本 不通过 AA
--text-muted / --bg-primary Aa var(--swatch-light-bg) 3.8:1 大号文本（18px+） 勉强通过
--accent (teal) / --bg-primary Aa var(--swatch-light-bg) 3.4:1 链接 / 可点击文本 不通过 AA
P0 --text-muted 在 Light 主题正文中对比度不达标
--text-muted (#6e7191) 在 --bg-primary (#f8f9fb) 上的对比度仅为 3.8:1，低于 WCAG AA 标准要求的 4.5:1（小文本）。这是 Light 主题下最常见的文字颜色之一，影响范围广泛。建议将 Light 主题的 muted 色加深至 #5c5f7a 左右（对比度 ≥ 4.5:1）。
P1 侧边栏图标按钮缺少 aria-label
侧边栏图标导航栏的按钮仅使用 title 属性提供提示，缺少 aria-label。屏幕阅读器用户无法理解这些图标的功能。状态指示器（.status-dot、.badge-dot）纯视觉表达，无文字替代。
P1 自定义弹出组件的键盘导航不完整
斜杠命令面板支持方向键和 Enter，但缺少 Escape 关闭、Tab 循环、Home/End 跳至首尾等标准键盘交互。MCP 服务器列表和技能列表不支持箭头键导航和 Enter 选择。
P1 resize 手柄无键盘替代
三栏布局的面板宽度调整和 SCM 面板的内部拆分器仅支持鼠标拖拽。需要添加键盘替代方案（如 Ctrl + 方向键调整面板宽度）。
P2 缺少高对比度模式
虽然提供了四套主题，但没有专门针对高对比度需求的主题选项。建议在设置中增加一个 "高对比度" 开关或主题预设。
维度四

视觉语言

SpaceCode 的视觉语言基础扎实：四套字体族选择精良（Space Grotesk、IBM Plex Sans、JetBrains Mono、Source Serif 4），玻璃态表面系统和辉光效果营造了现代感。但系统化程度仍有提升空间。

间距令牌系统化程度
30% — 未定义正式间距刻度
圆角令牌使用率
65% — 约 35% 组件仍硬编码 px 值
P1 间距值缺乏正式的令牌化刻度
当前的间距值是非正式的（4px、6px、8px、10px、12px、16px、20px、24px、32px 等），各组件自由选取，缺乏统一的间距刻度令牌。建议定义 --space-1 (4px) 到 --space-8 (32px) 的间距系统，在 _variables.scss 中集中管理。
P1 圆角和过渡令牌未被一致采用
设计令牌已定义 --radius-xs (4px) 到 --radius-xl (16px) 以及 --transition-fast/normal/slow，但约 35% 的组件仍使用硬编码的 border-radius: 8px、transition: all 0.2s ease 等值。
P2 字体大小设置项未全局生效
外观设置中允许选择字体大小，但全局 CSS 使用固定 14px 基准值，设置值存储后未通过 CSS 变量传播到所有组件。建议将字号设置绑定到 --font-size-base 变量。
P2 TitleBar 使用内联 SVG 而非 Lucide 图标
项目统一使用 lucide-vue-next 图标库，但 TitleBar.vue 中 VS Code 图标和 review-changes 图标使用了内联 SVG render 函数，风格与其他图标不一致。
P2 MessageItem.vue 中 CSS 块重复定义
.message-item.notification、.task-notification-card、.message-item 等 CSS 块在同一 <style> 区域重复定义 3-4 次（第 266-344 行和第 738-808 行），增加维护负担并可能产生优先级冲突。

P1 ChatPanel.vue 混合了过多关注点
ChatPanel.vue 超过 1000 行，同时处理会话管理、斜杠命令执行、差异查看、历史恢复、模型切换、消息处理。建议将会话管理和消息处理分离到独立 composable。
P1 组件对 Electron API 紧耦合
多个组件直接 import @/services/electronAPI，导致组件不可移植、难以进行单元测试。建议通过 provide/inject 或 Pinia action 层间接访问。
P1 i18n 覆盖率不完整 — 多处硬编码中文
项目已集成 vue-i18n，但 SCM 面板中有大量硬编码中文字符串（"提交"、"图形"、"合并冲突"、"取消暂存所有更改" 等），历史模态框和拖放覆盖层也绕过了 i18n 系统。
src/components/scm/ScmPanel.vue src/components/layout/ChatPanel.vue
P2 缺少共享基础组件库
每个功能区域独立发明按钮、表单输入和布局模式。Settings 组件有一套成熟的 s- 前缀组件系统（s-card、s-btn-primary、s-toggle-switch 等），但其他区域未能复用。
P2 SCM 文件列表缺少虚拟滚动
SCM 面板的文件变更列表和 Git 日志可能有数百条记录，当前全量渲染所有 DOM 节点。建议引入虚拟滚动（如 vue-virtual-scroller）优化性能。
