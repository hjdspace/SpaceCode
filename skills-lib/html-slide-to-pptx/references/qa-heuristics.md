# HTML → 可编辑 PPTX 常见翻车点 / QA Heuristics

这类转换最常见的问题，不是“生成失败”，而是**能生成，但细节悄悄炸版**。

## 一、文本类问题

### 1. 文本框高度不足 / 下边界溢出
典型表现：
- 段落最后一行被切掉
- 文本刚好压到圆角或边框
- 文本看似在 HTML 中没问题，但在 PPT 中换行后超出框

高风险原因：
- PowerPoint 的字体度量与浏览器不同
- 中文与英文混排导致换行位置漂移
- 文本框固定高度，但内容长度是动态的
- `fit: 'shrink'` / `resize` 并不可靠，很多情况下要打开 PPT 编辑后才会触发

### 2. 大标题压住下面的小字
典型表现：
- 标题和副标题、说明文字垂直间距不够
- 大标题一旦换到两行，就会压住下方说明

高风险原因：
- 标题高度按单行假设
- 标题字体太大或字体回退后字宽变大

### 3. 芯片 / pill / 小卡片中文字被裁切
典型表现：
- chip 里的字贴边
- 胶囊形状内上下边距不够
- 小卡片标题或正文多一两个字就超出

高风险原因：
- chip 高度太小
- 默认内边距未显式设置
- 文本缩放不可靠

## 二、布局类问题

### 4. 模块之间垂直间距累计漂移
典型表现：
- 上面看着都对，到了最下面开始打架
- 某个中间模块稍微变高，导致后续模块全部下移

高风险原因：
- 固定高度堆叠
- 某一块文案增长却没联动调整后续块坐标

### 5. 左右栏之间桥接箭头 / 连接关系偏位
典型表现：
- 箭头没对到中线
- 箭头方向反了
- 箭头在视觉上不居中

高风险原因：
- 连接点写死
- 调整列宽后没有同步修正箭头坐标

### 6. 边框、圆角、阴影与正文抢空间
典型表现：
- 内容没有越界，但视觉上已经太挤
- 文本距离边框太近，不像正式 PPT

高风险原因：
- 只检查“是否溢出”，没检查安全留白
- 圆角块实际可用区域比矩形更小

## 三、字体与渲染类问题

### 7. 字体回退导致字宽变大
典型表现：
- 代码里用 PingFang / 微软雅黑，但目标机器没有
- 打开 PPT 后自动回退到 Calibri / Arial / 其他字体，整体换行全变

### 8. 行高 / 段间距和 HTML 不一致
典型表现：
- HTML 看着松弛，PPT 里变挤
- 列表最后一行溢出

高风险原因：
- 浏览器 CSS 行高 ≠ PowerPoint 段落模型
- 默认段前段后距没显式控制

## 四、引擎已知限制

### PptxGenJS
- `fit: 'shrink'` / `resize` 并不等于生成时就自动完美缩放
- 文本框 margin、lineSpacing、fontFace 都需要尽量显式写死
- 复杂 HTML 不能直接当成浏览器布局来理解，只能语义映射

参考：
- PptxGenJS README 中对 `fit` 的说明
- GitHub issue: #544, #779, #991, #330

### python-pptx
- 对文本换行和字体度量控制更弱
- `word_wrap`、line spacing、顶部空白等行为也常有差异

参考：
- python-pptx issue: #709, #710, #162

## 五、建议加入 skill 的 QA 层

至少要检查：
1. 文本预计行数是否超过框高
2. 标题与副标题是否存在碰撞风险
3. bullet 列表是否超出安全高度
4. chip / 小卡片文字是否接近裁切阈值
5. 总体垂直堆叠高度是否超过容器高度
6. 箭头方向与连接位置是否符合 preset 规则
7. 是否依赖 `fit: 'shrink'` 这种不稳定机制

## 六、推荐 QA 分层

### A. 语义预审（现在就能做）
根据 preset 中的框大小、字号、文案长度估算风险。

### B. 结构审查（现在也能做）
检查生成对象坐标、间距、是否越界、是否重叠。

### C. 渲染复核（后续增强）
生成 PPT 缩略图后做热点检查，必要时接 OCR / 视觉模型。
